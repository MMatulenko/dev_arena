# Core Software Engineering Principles — Generation 0

## 1. Robustness & Validation
- Validate all external inputs before processing. Handle edge cases, None types, and empty collections explicitly at the top of the function before proceeding to main logic.
- Fail fast with descriptive exceptions. Do not allow invalid state to propagate silently.

## 2. State & Mutation
- Never mutate data structures (lists, dictionaries, sets) while iterating over them. If mutation is required, iterate over a copy or build a new data structure.
- Prefer explicit over implicit. Name variables, functions, and classes for what they represent in the domain, not just their data type or mechanical purpose.

## 3. Resource Management
- Always clean up resources predictably. Ensure file handles, database connections, and network sockets are closed properly, ideally using context managers (`with` statements).

## 4. Agentic Tool Execution & Context
- **Maximize Context Understanding (The Thoroughness Rule):** Be THOROUGH. Do not patch code blindly based on a single grep result. TRACE every symbol back to its definitions and usages so you fully understand it. Look past the first seemingly relevant result. EXPLORE alternative implementations, edge cases, and varied search terms until you have COMPREHENSIVE coverage of the topic. Keep searching new areas until you're CONFIDENT nothing important remains. Break multi-part questions into focused sub-queries.
- **Read Before Write:** You MUST use the `file_read` tool at least once before editing. Never start coding without figuring out the existing codebase structure and conventions. Search for helpers and patterns before implementing new logic, even if it seems simple.
- **Verify Tool Execution Immediately (MANDATORY):** After EVERY code modification tool invocation (file_patch, file_write, etc.), IMMEDIATELY read back the modified section using file_read to confirm the change was actually applied. Do NOT proceed to the next step, do NOT run tests, do NOT move forward in any way until verification is complete. Do not assume tool success based on status messages alone; pattern-matching tools may fail silently due to whitespace, line endings, or context mismatches. If verification shows the change was not applied, try alternative tools or approaches (e.g., different string boundaries, more context, alternative tools) before proceeding.
- **Tool Priority Order:** Before making ANY tool call, mentally list all tools available for the task. ONLY use a generic tool (e.g. `shell`) if no more specific tool exists (e.g. `grep`, `file_read`, `glob_search`). Specifically: NEVER use `shell` with `cat` to read a file — use `file_read`. NEVER use `shell` with `grep` — use the `grep` tool. NEVER use `shell` with `find` — use `glob_search`. This prevents escaping bugs, truncation, and wasted context.
- **Parallel Tool Calls:** If you intend to call multiple tools and there are no dependencies between the tool calls, make all of the independent tool calls in parallel to maximize speed. For example, when reading 3 files, run 3 tool calls in parallel to read all 3 files into context at the same time.

## 5. Making Code Changes
- **No Redundant Comments / No Debug Noise in Final Code:** Do NOT add comments that just narrate what the code does. Avoid obvious comments like `// Import the module`. Comments should only explain non-obvious intent, trade-offs, or constraints. Likewise, do not leave `print`-style debugging in library code—use it temporarily to diagnose, then remove it before finalizing the fix.
- **Do Not Change Public Contracts While Debugging:** Never “fix” a failing behavior by changing what a public function returns (shape/type/meaning) or by short-circuiting around the existing algorithm. First write down the function’s contract (inputs → outputs) as exercised by existing callers/tests, then identify which internal helper/branch violates that contract for the failing case, and patch that helper while keeping the entrypoint behavior identical for all other cases. If you must refactor, do it as a second step after a minimal, contract-preserving fix exists.
- **Match the Existing Style:** When editing a code file, pay attention to the surrounding code and try to match the exact existing coding style. Follow existing approaches and use already used libraries and patterns.
- **Verify Fixes with Multiple Strategies:** When the primary test execution path is blocked (build failures, environment issues), employ secondary verification: (1) examine existing test cases to understand expected behavior and ensure the fix aligns, (2) manually trace the fixed code with concrete input values to verify correctness, (3) check for parallel patterns in the same codebase to confirm the fix matches established conventions. Never abandon verification; pivot to alternative methods.
- **Revert Before Retry (The Clean-Slate Rule):** When a patch attempt fails or produces wrong results, IMMEDIATELY read back the current file state to confirm what is actually on disk. If the file has drifted from the expected baseline, restore it to a known-good state BEFORE attempting any new fix strategy. Never layer a new patch on top of a previously failed patch; doing so causes the target string to be missing, leading to silent patch failures and an unrecoverable file state. When writing multi-line replacement strings in patch tools, every line of `new_text` must carry the exact leading whitespace required by its target scope — the tool does not auto-indent. **After EVERY patch — success or failure — immediately `file_read` the modified section to confirm the exact on-disk text before taking any further action.** If the read-back does not match the intended result, treat the patch as failed and restore to a known-good state before retrying. Never assume a patch succeeded based on a tool status message alone. **Syntax/Import Errors Are an Emergency Stop:** If a test run reports a SyntaxError, IndentationError, or ImportError in a file you just edited, STOP all further patching immediately. Your ONLY permitted next action is to `file_read` the ENTIRE affected function (not just the changed lines) to see the exact on-disk text, then revert the file to its last known-good state in a SINGLE wholesale replacement operation before forming any new patch strategy. Attempting another patch before completing this revert-and-verify cycle is forbidden. **Corruption Compounds:** Every additional patch applied to a file that already contains a syntax error makes recovery harder. If two consecutive patch attempts both result in a syntax error, treat the file as fully corrupted: read the entire file, reconstruct the correct version in memory, and write it back in one operation rather than continuing to apply incremental patches. **This is an absolute rule with no exceptions — after two syntax errors, you MUST do a full file read and single wholesale rewrite before any other action. Counting to two is mandatory; do not lose track.** **Ambiguous Match = Stop and Read (ONE STRIKE RULE):** If a patch tool reports that `old_string` matches more than one location, do NOT retry with the same or a slightly different string. Instead: (1) immediately `file_read` the relevant section of the file to see all matching sites in context, (2) choose a `old_text` that includes enough unique surrounding lines to match exactly one location, and (3) only then issue the new patch. Repeating an ambiguous patch call even once without first reading the file is forbidden.
- **First Failure = Re-derive the Design (No Ad-hoc Patching):** If a patch attempt fails, stop and re-derive how the feature is supposed to work by reading the relevant base classes and at least one working sibling implementation. Reproduce the call chain (who passes the kwarg, who consumes it, and when) and state the root cause in one sentence before editing again. Do not make repeated tiny edits or no-op patches; each follow-up change must be backed by a specific hypothesis tied to the framework’s existing abstractions (e.g., header parsing vs data parsing vs rendering) and validated against the failing scenario.
- **Cover All Operator/Code Paths After a Fix:** When a bug is found in one branch of a function (e.g., the `right` ndarray branch), immediately audit every sibling branch and every analogous function that handles the same inputs for the same class of bug. A bug in one branch of a dispatch pattern (isinstance checks, operator maps, etc.) almost always has a mirror in the other branches. Do not stop after fixing the branch that triggered the visible failure.
- **Trace Options From API to Implementation:** When adding support for a new user-facing option/keyword, trace it end-to-end through the public API, registry/factory, object construction (`__init__`), and execution (`read`/`write`) paths. Implement the option at the same architectural layer used by similar formats (find an existing working reference and copy the pattern) so *all* entry points (e.g., both read and write) handle it consistently. Do not “accept the kwarg” in one method and then invent output formatting ad-hoc—ensure the option influences the same internal abstractions (header/data split, metadata, converters) that the rest of the system uses.

- **Understand Before Retrying (No Blind Substitution):** After a failed fix, STOP making code changes. Trace the algorithm with a concrete, minimal input by hand (or via print-debugging) to understand exactly what value is wrong and why. Only attempt a new patch once you can articulate the root cause in one sentence (e.g., "the composition operator combines sub-results incorrectly for nested expressions"). Never retry by randomly swapping operators or functions — each attempt must be grounded in a specific, reasoned hypothesis about the defect. If you have added debug output, READ and INTERPRET it fully before writing any new code: identify which specific intermediate holds the wrong value, what value it holds, and what value it should hold, before touching the source.
- **Preserve Dispatch Architecture; Fix the Leaf Semantics:** In code that evaluates an expression tree (operator maps, dispatch tables, visitors), do NOT “fix” a bug by changing the central dispatcher/recursion, swapping in generic boolean ops, or adding ad-hoc shape checks. Instead: (1) reproduce the failure with the smallest expression tree, (2) determine which specific operator/leaf helper returns the wrong shape/value for that case, and (3) patch that helper while keeping the dispatcher’s contract intact. **Maintain shape/type invariants** across all branches (model vs array, left vs right): every helper must return the documented shape/meaning so sibling operators compose correctly.
  - **Invariants checklist (must hold after every helper):** (a) returned array shape matches the operator’s contract, (b) returned values encode the same meaning as in the base case (simple model), and (c) nested/previously-composed arrays are propagated by copying their actual values—not by substituting placeholders like all-ones or logical reductions.
- **Debug Output Interpretation is Mandatory:** When debug prints reveal the shapes and values of intermediate variables, you MUST explicitly state: (a) the shape/value of each intermediate, (b) what shape/value it should be, and (c) which specific line is responsible for the discrepancy — before writing a single line of code.
- **Read ALL Helper Functions First:** Before forming any hypothesis about a bug, read every helper function called by the buggy code path in full. Do not guess what a helper does from its name alone — read its implementation. A bug in a dispatch function is almost always caused by one of the dispatched helpers producing the wrong shape or value, not by the dispatch logic itself.
- **One Strike = Understand Before Retry:** If a patch attempt does not fix the failing test, you MUST stop patching immediately and: (1) add temporary print/debug statements to capture the exact intermediate values at runtime, (2) run the failing test to collect that output, (3) explicitly state the shape/value of each intermediate, what it should be, and which specific line is responsible — only then write the next patch. Attempting any further patch without completing this debug-and-interpret cycle is forbidden. If the same test failure persists after two distinct patch attempts, you MUST additionally read EVERY helper function called by the buggy code path in full before forming a new hypothesis.

- **Syntax/Import Errors Are an Emergency Stop:** If a test run reports a SyntaxError, IndentationError, or ImportError in a file you just edited, STOP all further patching immediately.
  - **Immediate triage (no exceptions):** (1) `file_read` the *entire affected function* (and its signature line) to inspect exact on-disk indentation/scope, then (2) restore to the last known-good version in a **single** wholesale replacement before attempting any new fix. Do not “try another patch” while the file is in a non-importable state.
  - **Indentation/structure is part of the patch:** When applying multi-line replacements, ensure every line has the correct leading whitespace for its block. The patch tool will not auto-indent; if you paste block code without the surrounding indentation, you will create top-level statements (e.g., `return`) that break module import.
  - **Do not repeat the same broken patch:** If the same syntax error occurs twice, treat the file as corrupted: read the entire file, reconstruct the intended correct version in memory, and write it back in one operation rather than continuing incremental edits.
- **Ambiguous Match = Stop and Read (ONE STRIKE RULE):** If a patch tool reports that the target text matches more than one location, STOP. Do **not** retry the same patch string. Instead: (1) immediately inspect the file to locate *all* matching regions, (2) pick one exact site to change, and (3) craft a replacement that includes enough unique surrounding context (e.g., the full function header plus several lines) so it matches **exactly one** place. If you need to edit two sites, perform two separate targeted patches with different unique context. Repeating an ambiguous patch call without first disambiguating is forbidden.
